# [OPEN] ui-freeze-monitor

## Симптом
- Після останніх UI-правок фріз повернувся.
- За словами користувача, зависання схоже на попереднє: імовірно знову перевантажений `Programs/Edit` великою кількістю LVGL-віджетів або redraw.
- Поточний запит: запустити monitor і зібрати runtime-логи/stack trace без нових логічних змін до бізнес-коду.

## Гіпотези
1. `Programs/Edit` знову створює або одночасно тримає занадто багато LVGL-об'єктів, і UI task зависає в redraw/layout.
2. Scrollable список із 20 row widgets створює надто важкий tree для `480x320`, навіть якщо сам екран тепер масштабується правильно.
3. Проблема не в кількості row, а в каскадному relayout після переходу між екранами або після відкриття клавіатури.
4. Частина header/layout still invalidates великі області екрана, і watchdog знову спрацьовує в `lv_draw_label` або споріднених LVGL-функціях.
5. Поточний фріз відрізняється від попереднього і тепер сидить не в `Programs/Edit`, а в іншому екрані або callback під час навігації.

## План
- Запустити serial monitor.
- Відтворити фріз на платі.
- Зібрати повний WDT / panic backtrace.
- Зіставити лог із гіпотезами й лише після цього вносити мінімальні зміни.

## Статус
- Сесію відкрито.
- Отримано runtime-лог із monitor.
- Підтверджено: WDT виникає під час `create_screen_programs()` у `lv_label_create`, тобто під час масового створення віджетів `Programs/Edit`, а не в redraw після створення.

## Висновок за доказами
- Гіпотеза 1: підтверджена.
- Гіпотеза 2: підтверджена.
- Гіпотеза 3: не підтверджена поточним backtrace.
- Гіпотеза 4: не підтверджена поточним backtrace.
- Гіпотеза 5: відхилена, backtrace відповідає саме `Programs/Edit`.
