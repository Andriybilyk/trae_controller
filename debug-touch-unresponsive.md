# [OPEN] touch-unresponsive

## Симптом
- Після останньої прошивки користувач повідомив, що тач перестав реагувати.
- Регресія з'явилась після змін у `Programs/Edit`, де було збільшено кількість рядків кроків.
- Поки що немає доказу, чи тач не працює глобально, чи UI task зависає лише після входу в певний екран.

## Гіпотези
1. `Programs/Edit` знову створює занадто багато LVGL-віджетів на старті, через що UI task блокується і тач лише здається "мертвим".
2. Тач працює, але новий шар/об'єкт перекриває екран і перехоплює всі input events.
3. LVGL input loop живий, але після hard reset плата зависає в іншому місці ініціалізації, і UI не доходить до нормального event processing.
4. Регресія обмежена екраном `Programs/Edit`: на головному екрані тач живий, але на екрані програм після навігації події перестають оброблятись.
5. Проблема не в UI layout, а в runtime-стані після прошивки: monitor покаже panic/WDT або інший fail у UI/backend task.

## План
- Запустити `monitor` і зібрати стартові runtime-логи.
- Перевірити, чи є WDT/panic або ознаки зависання UI/touch loop.
- Лише після цього вирішити, чи потрібна інструментація, чи достатньо відкотити останню ризикову зміну.

## Доказ
- `Touch initialized` і `LVGL UI initialized` є в логах, отже стартовий UI/touch backend живі.
- Після натискання переходу в `Programs/Edit` користувач відтворив зависання.
- `task_wdt` підтвердив зависання `ui (CPU 1)`.
- Backtrace вказує на `create_screen_programs()` -> `lv_label_create()` на `screens.c:4244`.
- Отже корінь проблеми: не сам тач, а перевантаження побудови екрана `Programs/Edit`.
- Після спроби перейти на реальні рядки з поступовим добудовуванням `monitor` показав новий `task_wdt`.
- Новий backtrace вказує на `programs_step_row_build_one()` -> `lv_obj_create()` під час `screen_programs_refresh_rows()` у виклику з `tick_screen_programs()` ще в межах `create_screen_programs()`.
- Отже навіть "batch 2-3 rows" в тому ж циклі побудови екрана залишається завеликим для UI task.

## Рішення
- Додано мінімальну інструментацію побудови рядків `Programs/Edit`.
- Відкочено кількість одночасно створюваних рядків до малого безпечного значення для відновлення реакції UI.
- Поступове добудовування переведено в ще безпечніший режим: не більше одного рядка за тик і без примусового `tick_screen_programs()` всередині `create_screen_programs()`.
