# Session: program-save-time

## Симптоми
- Після натискання “ЗБЕРЕГТИ” нова програма/кількість кроків/час у UI не оновлюються одразу; оновлюються лише після перезавантаження.
- “Час випалу” рахується/зберігається некоректно.

## Гіпотези (фальсифіковані)
1) Дані зберігаються в NVS, але UI після “ЗБЕРЕГТИ” не оновлює RAM-модель/екран (екран списку створений один раз і не перебудовується).
2) Після “ЗБЕРЕГТИ” у UI залишається старий `selected_program_id`/`program_count`, тому tick/рендер бере старі значення.
3) `programs_update_info()`/парсинг значень кроків повертає некоректні temp/rate/hold, тому `total_time_s` рахується неправильно.
4) `ui_programs_persist_save()` викликається, але дані, які в нього потрапляють (steps/total_time/step params), неактуальні на момент виклику.

## План збору доказів
- Додати тимчасову інструментацію (ESP_LOGI) в:
  - `action_programs_save()` (pid/name/steps/total_time_s + факт виклику persist_save)
  - `programs_update_info()` (по кожному кроку: temp/rate/hold + вклад в час)
  - `create_screen_programs_list()` та tick (які значення малюємо одразу після save)

## Очікувані докази
- Якщо (1) правда: після save модель оновилась, але список не перемальовується/показує старе.
- Якщо (3) правда: в логах буде видно нульові/нереальні rate або ΔT, або “холд” не додається.
